Introduction

이번 프로젝트는 ‘인기 있는 강의’와 ‘비교적 인기가 덜 한 강의’의 비언어적 특성에 유의미한 차이가 있을까 라는 의문으로 시작했다.

온라인 강의를 제작할 때 기존의 인기있는 강의와 비교 분석해보고 싶은 부분이 있을 것이다. 해당 프로젝트에서 사용자는 영상 업로드 한번으로 비교 분석할 수 있는 결과를 얻을 수 있다.

이번 프로젝트에서 개인적인 목표는 AWS Infra를 직접 구축하고 제작한 웹과 모델을 배포해 보는 것, Docker를 활용해보는 것이었다.


Scheduling

팀원들 간 역할 분담 및 스케줄링은 WBS와 Kanban Board로 관리했다. 익숙하지 않은 스킬들이 들어가야 할 프로젝트이기에 학습과 스케줄링을 실시간으로 병행하기는 쉽지 않았다.

Kanban Board를 직접 활용해 본 것은 처음인데, 팀원들이 어떤 역할을 하고 있는지와 앞으로 해야 할 것들에 대한 Board들을 한 눈에 볼 수 있어서 꽤 유용했다.

처음 시작할 때 카테고리는 [시작 전], [진행 중], [완료] 3가지로 나눠서 작업했지만 계속해서 늘어나는 보드들과 한 눈에 보기 어려워졌다. [완료된지 1주 이상 경과], [보류], [백로그] 탭을 추가함으로써 관리하기 수월해졌다.

처음 웹에 접근했을 때는 사용자가 Video를 업로드 할 수 있는 화면이 나오고,
업로드 후에는 설정한 Feature마다 비교 분석 할 수 있는 그래프가 나온다.

Web 기능 정의서

사용자는 해당 플랫폼을 통해 어떤 행동을 해야하고 얻어갈 수 있는지, 개발자는 어떤 부분들을 생각하며 작업해야 하는지에 대해 기능 정의서를 먼저 작성하고 작업에 들어갔다. 물론 처음에 적은 정의서로부터 계속 수정이 필요했다.


Text Similarity

이번 프로젝트는 비언어의 특성 분석이 메인 주제였지만, 기획한 결과 비언어적 특성이 생각보다 많이 없었다. 따라서 비언어적 특성에 더불어 나는 언어적 특성을 추가로 분석하기로 했다.

각 강의별 텍스트를 비교하는 대상은 ‘국가교육과정 정보센터’에서 볼 수 있는 ‘수학 성취 기준’이다. 성취도와 비교한 이유는 공식적으로 정의된 것과 해당 수업의 내용이 얼마나 유사한지 비교함으로써 하나의 객관적 지표로 활용될 수 있을 것이라 생각했기 때문이다.

언어적 특성은 사용자가 Video를 업로드했을 때 자동적으로 음성을 분리하고 텍스트로 만들어주는 과정을 거친다. 텍스트가 만들어지면 nltk 패키지로 전처리를 하고, langchain과 GPT API로 요약을 진행한다.
요약 과정을 거친 이유는 방대한 강의 내용 축소와 API 사용 경험을 향상시키기 위함이었다.

GPT API는 $5이내에서 무료로 사용할 수 있다. (가입 후 3개월 이내)

RPD(Request Per Day), RPM(Request Per Minute) 제한 문제가 있었는데 RPD는 Google ID를 추가 생성하여 해결할 수 있었고, RPM은 time 패키지를 활용하여 요약 프로세스를 문제 없이 자동화 할 수 있었다.

텍스트 유사도는 두 개 이상의 문서나 문장 간의 유사성을 계산하는 기술이다. 이를 위해 비교하고자 하는 문서를 벡터로 나타내야 하는데, 텍스트를 벡터화하기 위해 이 프로젝트에서 사용한 Embedding 방법은 비교적 구현이 쉽고 빠른 TF-IDF를 사용했다.

유사도를 측정하기 위해서도 다양한 방법이 있는데, 그 중 벡터간 유사도를 활용하는 Cosine Similarity와, 벡터 간 거리를 활용하는 Euclidean Distance를 활용했다.

측정 결과 ‘인기 강좌’의 Cosine Similarity가 ‘상대적으로 인기가 덜 한 강좌’보다 높고, Euclidean Distance는 더 가깝다는 결과가 나왔다. (’인기 강좌’가 ‘성취 기준’과 더 유사하다).

이번 프로젝트에서 비교할 수 있는 강좌가 3가지 밖에 없어서 통계적 유의성을 검증하여 설득력을 높이기에는 어려움이 있었다. 만약 여러 강좌를 비교하여 유사도를 측정할 수 있는 지표를 나타낸다면, 성취 기준과의 유사도가 인기도를 판단할 수 있는 지표일지 생각할 수 있을 것이다.


AWS Architecture Develop

참 많이 봤지만 처음으로 직접 그려본 Architecture다.

머리 속에서 추상적으로 있던 server와 database, web의 상호작용을 시각적으로 그려보니 훨씬 이해하기도 쉬웠고 어떤 부분을 수정해야 할 지 파악하기 좋았다.

두 번째 그려본 Architecture.

S3의 Trigger라는 개념을 알게 됐고, Lambda를 구동하여 Model을 실행해 볼 수 있겠다는 생각을 했다. 하지만 아직 Web이 AWS Cloud 칸 밖에 존재한다. Plotly Dash Web은 어디에 위치해야 할지 아직 감을 못 잡은 듯 하다.

어느 정도 Architecture에 대한 이해를 하고, Modeling 결과를 받는 데 까지 시간이 걸린다는 것을 인지하고 AWS SQS 서비스를 추가한 세 번째 Architecture.

개발 단계에서 사용자 데이터와 비교할 데이터를 적재하는 과정과 Docker를 어떤 부분에서 사용해야 할지 함께 그리기 시작했다.

최종 구현 완료된 Architecture.

다른 팀원들이 분석에 활용한 Model은 같지만 다른 parameter를 활용했는데, 병렬 구조로 실행하려면 AWS 내 Kubernetes를 사용하는 것이 일반적이라고 들었다. 하지만 프로젝트 기간 내 구현하기 어려움이 있어 컨테이너를 쉽게 관리할 수 있는 ECS를 활용했고 Lambda로 ECR에 올려진 Image들을 구동함으로써 병렬 처리가 가능하게 했다.

Trigger는 S3에 영상이 적재됐을 때를 활용했는데, Lambda를 여러 개 사용하지 못하는 문제점을 발견하고 하나의 Lambda에 두 개의 Image를 실행하는 코드로 변경했다.


# 환경변수 설정

import boto3
import os

s3 = boto3.client('s3', 
                   aws_access_key_id = os.environ.get("AWS_ACCESS_KEY_ID"), 
                   aws_secret_access_key = os.environ.get("AWS_SECRET_ACCESS_KEY"))
bucket_name_video = os.environ.get("BUCKET_NAME_VIDEO")
bucket_name_text = os.environ.get("BUCKET_NAME_TEXT")
AWS Service를 이용하기 위해서는 Key 값들이 필요한데, 보안성을 위해 Key가 포함된 코드를 Docker Image로 만들지 않고 AWS 내에서 환경 변수로 설정해줬다.
